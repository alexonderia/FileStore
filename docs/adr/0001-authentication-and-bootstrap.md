# ADR 0001: аутентификация и создание первого суперадминистратора

- Статус: принято
- Дата: 2026-07-16

## Контекст

FileStore требует саморегистрацию пользователей, вход по существующей учетной записи, отзывные API-учетные данные и безопасный способ создать первого глобального администратора. Автоматическое повышение первого публично зарегистрированного пользователя создает гонку и позволяет случайному посетителю получить административный доступ.

## Решение

- Пользователи регистрируются и входят по нормализованному email и паролю.
- Пароли хранятся только в виде современного адаптивного hash; конкретный алгоритм и параметры фиксируются перед ревью миграций этапа 1.
- При login выдается непрозрачный Bearer token с высокой энтропией. В PostgreSQL хранится только его hash и метаданные жизненного цикла.
- Первый суперадминистратор создается или повышается локальной идемпотентной deployment-командой `filestore-api bootstrap-superadmin`.
- Bootstrap не публикуется как HTTP endpoint и не хранит учетные данные в миграциях или исходном коде.
- Публичная регистрация всегда создает обычного пользователя. Пользователь никогда не становится суперадминистратором из-за порядка регистрации.
- Пароли, сырые токены и bootstrap-секреты маскируются в логах и диагностике.

## Последствия

Этап 1 требует hashing паролей и токенов, отзыва токенов, rate limiting на границе развертывания и интеграционных тестов для конкурентной регистрации и bootstrap. Сброс пароля и внешние identity-provider остаются вне MVP.
