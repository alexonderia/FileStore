# ADR 0003: транзакции PostgreSQL и объектное хранилище

- Статус: принято
- Дата: 2026-07-16

## Контекст

PostgreSQL и SeaweedFS не могут участвовать в одной атомарной транзакции. Конкурентные update session и hard lock также затрагивают несколько таблиц сразу.

## Решение

- PostgreSQL остается источником истины для метаданных, прав доступа, версий, сессий, ссылок и блокировок.
- Байты опубликованных объектов неизменяемы, а object key никогда не переиспользуется.
- Длинные запросы к SeaweedFS никогда не выполняются внутри транзакции PostgreSQL.
- Потоки загрузки сначала записывают объект, затем фиксируют метаданные. Если commit не удался, выполняется best-effort удаление и последующая сверка orphan-объектов.
- Любая запись в файл блокирует строки в порядке `files`, затем зависимая строка session/lock/link.
- Ожидаемые конкурентные конфликты обеспечиваются ограничениями базы данных и маппятся в доменные ответы.
- Создание файла атомарно создает версию 1 и необходимые current/version link-записи после появления миграции ссылок.
- Resolve атомарно создает опубликованную версию, ее version-link, обновляет current version и завершает session.
- Cleanup сначала фиксирует терминальное состояние в базе, а уже потом удаляет несвязанные candidate-байты с retry и льготным периодом.

## Последствия

Временные orphan-объекты считаются ожидаемым и восстановимым состоянием. Интеграционные тесты должны уметь внедрять сбои на каждой границе между базой и object storage, а также проверять конкурентные переходы session и lock.
